# Windcode 扩展运行时 Spec

## 背景

Windcode 已具备统一事件内核、模型适配、工具注册、权限与沙箱、会话恢复、上下文压缩、子智能体和 Worktree 编排，但扩展能力仍主要依赖核心代码和静态配置。MCP、Skills、Hooks 和插件资源尚未形成统一的发现、信任、装配、运行与观测边界。

本阶段融合五类架构的长处：

- Claude Code：可组合扩展和人类控制。
- Codex：声明式能力包、来源绑定、解析与激活分离。
- Pi Agent：小内核、生命周期事件和按需加载。
- GenericAgent：高信息密度与避免扩展 Schema 持续占用上下文。
- Hermes Agent：Provider 生命周期、故障隔离和跨入口一致性。

整体路线拆为三个独立实施周期：

1. 扩展运行时：MCP、Agent Skills、Hooks、插件发现和本地安装。
2. 插件生命周期：Git 安装、锁定、更新、回滚、卸载和市场 Provider。
3. 长期记忆：用户画像、经验结晶和受控自我进化。

本 Spec 只覆盖第一个周期。

## 目标

- 建立统一扩展运行时，使内建能力与外部能力复用现有工具、事件、权限、沙箱、取消和审计机制。
- 支持通过 stdio 和 Streamable HTTP 接入 MCP Server，并使用其 Tools、Resources、Prompts 和 Instructions。
- 支持符合 Agent Skills 规范的技能包，兼容发现 Claude Code、Codex 和 Pi 常见技能目录，并按需加载正文。
- 提供稳定的高层生命周期 Hooks；Hook 可以观察、通知、追加带来源的上下文或拒绝操作，但不能提升权限。
- 支持版本化的 Windcode 声明式插件清单，使本地插件能够组合 Skills、MCP、Hooks、命令和权限需求。
- 提供用户级、项目级和运行级资源作用域；项目扩展只有在工作区受信任后才能激活。
- 让 CLI、TUI 和公共 SDK 通过同一服务层发现、检查、启禁和重载扩展。
- 在扩展缺失、连接失败或执行异常时保持故障隔离，并提供结构化、可操作的诊断。
- 控制启动与上下文成本：只扫描元数据、按需连接 MCP、按需激活大型工具集。
- 为后续完整插件生命周期和长期记忆预留稳定边界，但不提前实现其行为。

## 功能需求

- F1：系统提供统一的扩展能力目录，能够描述能力的名称、类型、来源、作用域、启用状态、信任状态、权限需求和诊断状态。
- F2：扩展资源按内建、用户级、项目级、本次运行显式指定的顺序组合。高优先级独立资源可以按明确规则覆盖低优先级资源；插件组件不得被静默覆盖。
- F3：项目级 Skills、Hooks、MCP 配置和插件只有在当前工作区已被用户信任后才能激活。未信任资源可以被列出和检查，但不得执行或向模型注入内容。
- F4：系统支持版本化的 Windcode 声明式插件清单。清单能够声明插件元数据、兼容版本、Skills、MCP Servers、Hooks、命令、权限需求和插件数据需求。
- F5：插件具有稳定唯一标识，插件内组件使用插件命名空间。清单路径不得逃逸插件根目录；无效路径、重复标识、未知必需字段和不兼容版本必须产生明确诊断。
- F6：系统能够从本地目录发现、校验和安装声明式插件，并记录其来源和内容摘要。安装前不得执行插件中的命令、脚本或 MCP Server。
- F7：插件声明的命令只能安全地路由到声明式提示、Skill 或已有能力入口；首版不允许命令条目直接携带任意宿主代码。
- F8：安装、启用、禁用或配置变化后，用户可以显式重载扩展。每次运行使用不可变能力快照，重载不得暗中改变正在进行的运行。
- F9：系统支持通过 stdio 和 Streamable HTTP 连接 MCP Server。连接具有明确超时、取消、关闭和失败恢复语义，不得遗留子进程、连接或后台任务。
- F10：MCP Server 的 Tools、Resources、Prompts 和 Instructions 均可被发现和使用，并保留服务器来源。Resources 和 Prompts 通过受控运行时接口按需获取，Instructions 以有界、可识别来源的内容进入上下文。
- F11：首版 MCP 认证支持环境变量引用和静态请求头引用；密钥不得以明文进入普通配置、事件、日志、模型上下文或错误信息。OAuth 和动态客户端注册不在本阶段实现。
- F12：MCP Tool 必须转换为统一工具语义，复用参数校验、权限策略、调度、取消、事件、结果截断和审计能力。MCP Server 提供的风险注解仅作为提示，不构成信任依据。
- F13：未知 MCP Tool 默认被视为可能产生外部副作用并进入审批流程。用户可以按服务器和工具配置固定策略，但配置只能在现有权限上限内授权，不能绕过沙箱或运行级限制。
- F14：少量 MCP Tools 可以直接进入模型工具目录；超过配置阈值时，只提供精简能力目录和搜索入口，完整工具描述按需激活，避免永久占用上下文。
- F15：系统按照 Agent Skills 规范发现并校验技能包，同时兼容用户显式配置的 Claude Code、Codex 和 Pi Skills 目录。项目级 Skills 仍受工作区信任约束。
- F16：Skills 采用渐进披露：系统常驻内容只包含必要元数据；完整指令、引用、脚本和资源仅在技能匹配或用户显式调用后读取。Skill 内容和辅助文件必须保留来源边界。
- F17：同名独立 Skill 按确定的作用域优先级选择，并向用户显示覆盖关系；同一优先级发生冲突时必须报错，不得依赖不稳定的扫描顺序。
- F18：Hooks 支持会话启动与结束、用户提交、运行开始与结束、工具调用前后、权限请求、上下文压缩前后、子智能体启动与结束以及运行错误等高层事件。
- F19：Hook 可以执行通知、受控命令或上下文提示动作。预执行 Hook 可以拒绝或收紧操作；后置 Hook 只能观察、通知或追加带来源的上下文，不能篡改历史结果或提升权限。
- F20：所有 Hook 命令都必须经过现有策略、沙箱、超时、输出限制、取消和审计流程。Hook 失败不得自动允许原本被拒绝的操作，也不得无限重试。
- F21：系统隔离单个扩展的发现、启动和运行故障。可选扩展失败时，其他能力和 Windcode 核心仍可使用；标记为必需的扩展失败时，相关启动过程必须明确终止。
- F22：每个插件使用独立数据目录。插件配置可以持久化，密钥只能引用环境变量或外部凭据存储；插件不得通过清单取得未声明的持久化位置。
- F23：CLI、TUI 和公共 SDK 通过同一管理行为列出、检查、本地安装、启用、禁用和重载扩展，并展示一致的状态、来源、权限需求和诊断信息。
- F24：扩展发现、连接、激活、Hook 执行、MCP 调用、权限判定、失败和重载都产生可关联的结构化事件，并进入现有会话与本地轨迹体系。
- F25：未配置或未启用扩展时，现有单智能体、多智能体、工具、会话、权限、沙箱、CLI、TUI 和 SDK 行为保持不变；旧配置和旧会话无需迁移即可读取。

## 非功能需求

- N1：首版继续以 Linux 为正式支持和验收平台；其他平台可以运行，但不作为本阶段兼容承诺。
- N2：未配置扩展时，系统不得启动额外进程、建立网络连接或扫描无关目录，启动和运行行为应接近当前版本。
- N3：扩展发现只读取必要元数据。MCP 连接采用按需启动或有界并发，不得因单个慢速 Server 无限阻塞启动、重载或退出。
- N4：插件清单、Skill 内容、Hook 输出、MCP Schema、Resources、Prompts 和 Instructions 均视为不可信外部输入，必须经过大小限制、格式校验、来源标记和边界检查。
- N5：扩展资源不得绕过现有权限、审批、沙箱、工作区边界、网络限制、运行预算或取消机制；任何配置组合都不得产生隐式权限提升。
- N6：插件根目录、组件路径、数据目录和 Skill 辅助文件必须防止路径逃逸、符号链接越界和跨插件数据访问。
- N7：发现插件或加载元数据不得执行插件代码、启动 MCP Server、运行 Hook、展开 Skill 脚本或访问未声明网络地址。
- N8：扩展、Hook 或 MCP Server 在启动、执行、取消和关闭后不得遗留子进程、网络连接、文件句柄、异步任务或未完成的审批等待。
- N9：相同配置和文件状态必须产生稳定的发现顺序、覆盖关系、能力标识和诊断结果，不得依赖文件系统遍历顺序或并发完成顺序。
- N10：单个可选扩展的超时、崩溃、协议错误、无效输出或资源耗尽不得破坏其他扩展、进行中的核心运行或会话持久化。
- N11：MCP 和 Hook 输出必须有界。超出限制的完整内容可以进入本地 artifact，但模型视图和终端默认展示必须使用可解析摘要与引用。
- N12：大型扩展目录和 MCP 工具集不得线性膨胀常驻模型上下文；只有完成当前任务所需的元数据和已激活能力进入模型视图。
- N13：旧配置、旧会话记录和现有公共开发接口保持兼容；新增配置字段具有安全默认值，新增事件具有明确版本，旧消费者可以忽略未知事件。
- N14：CLI、TUI 和 SDK 对同一扩展操作必须产生一致的状态转换和事件语义，不得维护彼此独立的扩展状态。
- N15：扩展状态、诊断和事件默认只保存在本地，不新增遥测或远程同步；密钥、认证头和已识别敏感值必须在日志、事件和错误中脱敏。
- N16：错误信息必须包含失败阶段、扩展或服务器来源、错误类别和可操作建议，同时避免泄露密钥、完整敏感路径或未经截断的外部输出。
- N17：自动化测试不得依赖真实模型凭据、外部网络、用户现有插件或用户 MCP 配置；stdio 和 Streamable HTTP 场景必须能由本地模拟 Server 稳定复现。
- N18：真实远程 MCP 和第三方插件测试只能作为显式启用的冒烟测试，不得进入默认测试套件。
- N19：扩展运行时必须保持清晰的单向依赖边界，使 MCP、Skills、Hooks 和插件管理不依赖 TUI，并允许后续插件分发和长期记忆复用同一能力目录与生命周期事件。
- N20：格式检查、静态类型检查、单元测试、协议契约测试、集成测试、端到端测试和构建必须在无模型密钥环境中通过。

## 不做的事

- 不在本阶段实现 Git、压缩包 URL 或在线市场插件安装。
- 不实现插件锁文件、远程版本检查、更新、回滚、完整卸载、市场发布或依赖解析；这些属于第二个 Spec。
- 不实现后台自动安装、自动启用或自动更新插件。
- 不实现任意进程内 Python 插件、动态 Python 模块导入或其他拥有宿主完整权限的扩展代码。
- 不实现独立插件进程与通用插件 RPC；只为后续受隔离的可执行插件预留架构边界。
- 不实现插件提供自定义 TUI 控件、主题、模型 Provider、执行后端或新的公共 SDK 模块。
- 不实现 Hook 的直接 HTTP 请求、启动子智能体、替换工具结果、修改历史消息或放宽权限。
- 不向插件暴露模型流式增量、底层存储写入和其他易变的内部事件。
- 不实现文件变化后自动热重载；扩展变化只通过显式重载影响后续运行。
- 不允许重载改变正在进行的运行所持有的能力快照。
- 不实现 MCP OAuth、动态客户端注册、Elicitation、Sampling 或将 Windcode 暴露为 MCP Server。
- 不承诺支持所有 MCP 可选协议扩展；首版只承诺 Tools、Resources、Prompts、Instructions 及其必要生命周期。
- 不将 MCP Server 的风险注解、插件来源或工作区信任等同于工具执行授权。
- 不把全部 MCP 工具描述或 Skill 正文永久注入模型上下文。
- 不实现跨会话长期记忆、用户画像、经验结晶、Skill 自动演进、向量检索或后台模型反思；这些属于第三个 Spec。
- 不改变现有模型协议、Agent Loop、多智能体拓扑、Worktree 集成、安全模式或会话恢复语义。
- 不自动迁移、重写或删除现有配置、会话、轨迹和用户扩展数据。
- 不直接复制 Claude Code、Codex、Pi Agent、GenericAgent、Hermes Agent 或 mewcode 的源码；只采用经过验证且适合 Windcode 的架构思想和行为模式。
- 不把 macOS、原生 Windows、远程执行集群或企业多租户作为本阶段正式验收范围。

## 验收标准

- AC1（F1、F2）：配置内建、用户级、项目级和运行级扩展后，三个入口展示相同的能力目录、来源、作用域、状态和覆盖关系；重复运行结果顺序一致。
- AC2（F3）：未信任项目中的 Skills、Hooks、MCP 和插件可以被检查但不会激活、执行或注入上下文；用户建立信任并显式重载后，新运行才能使用这些能力。
- AC3（F4、F5）：有效声明式清单能够组合 Skills、MCP、Hooks、命令和权限需求；路径逃逸、符号链接越界、重复插件 ID、组件冲突、不兼容版本和未知必需字段均被拒绝并给出来源明确的诊断。
- AC4（F6）：本地插件安装会完成发现、校验、来源记录和内容摘要生成；在安装与检查阶段，插件脚本、Hook 命令和 MCP Server 均未被执行或启动。
- AC5（F7）：插件声明的命令能够触发对应提示、Skill 或已有能力；尝试通过命令清单直接执行任意宿主代码时，清单校验失败。
- AC6（F8）：运行开始后禁用插件并执行重载，原运行仍使用原能力快照；重载之后创建的新运行使用新快照，状态和事件可区分两个版本。
- AC7（F9）：模拟 stdio 与 Streamable HTTP MCP Server 均能连接、取消和关闭；连接超时、协议失败及运行中取消后，无遗留子进程、连接或后台任务。
- AC8（F10）：模拟 MCP Server 暴露 Tool、Resource、Prompt 和 Instructions 后，系统能够分别列出并按需使用它们；进入上下文的内容带有服务器来源且受大小限制。
- AC9（F11）：环境变量和静态请求头引用能够完成 MCP 认证；配置、事件、轨迹、错误和模型上下文中不出现对应密钥或完整认证头。
- AC10（F12）：MCP Tool 能够复用统一参数校验、调度、权限、取消、进度、结果截断和审计行为；参数错误、远端错误和取消均形成结构化工具结果。
- AC11（F13）：未配置规则的 MCP Tool 被视为有外部副作用并触发审批；服务器将其标记为只读也不会自动放行。显式工具策略可以减少重复审批，但不能突破当前权限和沙箱上限。
- AC12（F14）：MCP 工具数量低于阈值时，工具可直接使用；超过阈值时，常驻上下文只包含精简目录，模型通过搜索激活目标工具后才能取得完整描述并调用。
- AC13（F15、F16）：来自 Windcode、Claude Code、Codex 和 Pi 配置目录的有效 Agent Skill 均可发现；初始上下文只包含必要元数据，匹配或显式调用后才读取完整 Skill 及其引用资源。
- AC14（F17）：不同作用域出现同名独立 Skill 时，系统按确定优先级选择并展示覆盖关系；同一作用域内重复名称产生明确错误，改变文件遍历顺序不会改变结果。
- AC15（F18）：模拟一次完整运行时，可以观察到会话、用户提交、运行、工具、权限、压缩、子智能体和错误相关 Hook 事件在对应生命周期点触发。
- AC16（F19、F20）：预执行 Hook 可以拒绝危险工具或增加限制，拒绝后不产生工具副作用；后置 Hook 可以通知或追加来源明确的上下文，但不能修改工具结果或提升权限。Hook 命令同样经过审批、沙箱、超时、取消和输出限制。
- AC17（F20、F21）：可选 Hook、Skill、插件或 MCP Server 启动及运行失败时，故障被隔离，其他扩展和核心 Agent 可继续运行；必需扩展失败时，启动明确终止并显示失败来源和修复建议。
- AC18（F22）：两个插件的配置和持久数据彼此隔离，路径逃逸和跨插件访问被拒绝；密钥只能通过安全引用取得，普通配置中写入明文密钥时校验失败。
- AC19（F23）：同一个本地插件可以分别通过 CLI、TUI 和 SDK 完成列出、检查、安装、启用、禁用和重载，三者观察到一致的状态转换与诊断结果。
- AC20（F24）：扩展发现、连接、激活、Hook、MCP、权限、失败和重载事件均含可关联的会话、运行、插件或服务器来源信息，并先持久化再向消费者发布。
- AC21（F25）：不配置任何扩展时，现有单智能体和多智能体端到端场景、旧配置读取、旧会话恢复、权限、沙箱、TUI 和 SDK 测试保持通过，且启动期间没有新增网络连接或子进程。
- AC22（N3、N8、N10、N11）：慢速、崩溃、超大输出和错误协议的模拟扩展均受到超时与资源边界约束；取消或退出后资源完成清理，完整大输出进入本地 artifact，模型与终端只接收摘要引用。
- AC23（N4、N5、N6、N7、N15、N16）：恶意清单、路径、Skill、Hook 输出和 MCP 内容不能逃逸目录、执行安装期代码、扩大权限或泄露密钥；用户能从结构化诊断定位失败阶段和来源。
- AC24（N9、N12、N14）：对同一组扩展重复发现和重载，能力标识、排序与冲突结果稳定；大型能力集不全部进入常驻上下文；CLI、TUI 和 SDK 不产生分叉状态。
