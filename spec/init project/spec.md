# Windcode 纵向 MVP Spec

## 背景

现有 Coding Agent 各有明显优势：Claude Code 强调扩展性和人类控制，Codex 强调受控执行与可验证变更，Pi 强调小内核和可组合性，GenericAgent 强调高信息密度上下文与经验检查点，Hermes Agent 强调跨模型韧性和持续运行。

Windcode 首期不机械复制五套系统，而是构建统一的分层事件内核，交付一个可实际完成本地编码任务、可嵌入其他应用、执行过程透明且受用户控制的纵向 MVP。

## 目标

- 支持从自然语言任务到代码检查、修改、命令执行和结果验证的完整闭环。
- 终端交互界面和嵌入式开发接口共享同一套 Agent 行为。
- 原生支持三类主流模型协议，并允许显式跨模型故障恢复。
- 在长任务中保持上下文连续性、会话可恢复性和执行安全性。
- 建立稳定但最小的扩展边界，为后续多智能体、MCP、Skills 和长期记忆演进提供基础。

## 功能需求

- F1：提供聊天流式终端工作台，展示增量回复、推理状态、工具调用、审批请求、错误、用量和最终结果。
- F2：提供可嵌入式开发接口；外部程序能够提交任务、消费结构化增量事件、响应审批并取消运行，其行为与终端工作台一致。
- F3：支持 Anthropic Messages、OpenAI Responses 和 OpenAI-compatible Chat Completions，并将三者转换为一致的内部消息与事件语义。
- F4：Agent 能持续执行“模型推理—工具调用—结果反馈”循环，直至任务完成、用户取消、预算耗尽或发生不可恢复错误。
- F5：内置文件读取、文件写入、精确编辑、补丁应用、文件匹配、文本搜索、Shell 执行和询问用户工具；参数错误和执行错误必须作为结构化结果反馈给 Agent。
- F6：只读工具可安全并行；可能修改工作区或产生外部副作用的工具必须按顺序执行，且工具结果顺序保持确定。
- F7：提供计划、默认、自动编辑和完全授权四种权限模式；模式切换、审批、拒绝和执行结果必须可见并可审计。
- F8：在支持系统隔离的 Linux 环境启用工具沙箱；无法启用时必须明确告知用户并收紧审批，不得静默降级。
- F9：持续保存会话事件，支持重启恢复、回退到历史节点及从历史节点创建分支；异常退出后不得自动重复未确认的副作用操作。
- F10：按层级加载仓库持久指令，以 `AGENTS.md` 为规范入口，并兼容 Windcode、Claude 和 Hermes 的项目指令文件；离当前工作目录更近的规则具有更高优先级。
- F11：在上下文接近预算时，先裁剪低价值的大型工具结果和历史媒体，再生成包含任务目标、关键决策、相关文件、当前进度、未完成事项和验证证据的检查点摘要，同时保留最近交互。
- F12：对限流、短暂网络失败和服务端错误执行有限重试；重试失败后仅按用户预先配置的回退链切换模型，且每次切换必须明确展示。
- F13：支持用户级和项目级配置叠加，项目级配置优先；密钥只从外部安全来源读取，配置错误必须给出可操作的诊断信息。
- F14：允许外部代码注册自定义工具和模型传输适配器，并复用内置参数校验、事件、取消和权限机制。
- F15：记录仅保存在本地的结构化运行轨迹，包括事件时间线、耗时、模型用量、工具状态和错误；敏感信息必须默认脱敏。
- F16：任务完成时汇总工作区变更、已运行的验证及其实际结果；未验证或失败的内容必须明确说明，不得宣称成功。

## 非功能需求

- N1：首期以 Linux 为正式支持和验收平台。
- N2：终端界面在模型流式输出和长时间工具运行期间保持可交互，用户取消应及时传播到模型请求和子进程。
- N3：会话、配置和轨迹写入应具备异常中断恢复能力，不留下被误判为成功的半完成状态。
- N4：默认不上传遥测、会话、代码、工具参数或诊断轨迹。
- N5：公共开发接口具备严格、稳定且可发现的契约；终端界面不得绕过公共核心行为。
- N6：自动化测试不得依赖付费模型密钥；真实模型冒烟测试仅在显式提供凭据时运行。
- N7：项目以 Apache-2.0 许可证发布。
- N8：首期性能目标以本地交互流畅、事件无丢失、工具取消可靠为准，不以高并发多租户吞吐为目标。

## 不做的事

- 不实现多智能体、子智能体、团队协作或工作树并行编排。
- 不实现 MCP、Skills、Hooks、插件发现、插件安装或完整插件生命周期。
- 不实现跨会话长期记忆、用户画像、经验结晶或自我进化。
- 不实现浏览器控制、网页搜索、图像生成或专用 Git 工具。
- 不实现远程执行后端、消息网关、桌面端、Web 界面或 IDE 集成。
- 不提供正式的单次 Print 或 JSONL 命令行产品入口。
- 不承诺首期支持 macOS、原生 Windows 或企业多租户场景。
- 不静默选择模型、扩大权限、启用网络或执行工作区外写入。
- 不直接复制参考项目源码；只借鉴经过验证的架构思想和行为模式。

## 验收标准

- AC1（F1、F2）：同一个模拟编码任务可分别通过终端工作台和嵌入式接口运行，两者产生语义一致的文本、工具、审批、用量和完成事件。
- AC2（F3）：三类协议的模拟服务分别返回流式文本、推理信息、多个工具调用、用量和错误时，系统均生成一致的内部事件。
- AC3（F4）：Agent 能完成“读取文件—修改代码—运行测试—报告结果”的完整模拟任务，并在达到迭代预算时可靠终止。
- AC4（F4、N2）：用户取消模型请求或长时间 Shell 命令后，运行停止且会话记录为已取消，不继续执行后续工具。
- AC5（F5）：八种内置工具均有成功、参数错误和执行失败场景，结果能够继续反馈给模型。
- AC6（F6）：多个只读调用能够并行完成；写入和命令调用保持顺序，返回结果与模型调用顺序一致。
- AC7（F7）：四种权限模式均可观察验证；拒绝审批后不产生对应副作用，Agent 能收到拒绝结果。
- AC8（F8）：沙箱可用时，越界写入和未授权网络访问被阻止；沙箱不可用时，界面显示降级状态且高风险操作需要审批。
- AC9（F9）：进程重启后能够恢复消息、工具状态和上下文；可从历史节点回退并创建不覆盖原历史的新分支。
- AC10（F9、N3）：在副作用工具执行前后模拟异常退出，恢复后不会把未完成操作标记成功，也不会自动重复执行。
- AC11（F10）：根目录和嵌套目录存在不同指令时，发送给模型的有效规则符合就近优先；兼容文件冲突时采用可预测的主文件优先级。
- AC12（F11）：超出上下文阈值的长会话能够压缩并继续任务；压缩后仍保留目标、关键文件、决策、进度和验证证据，原始审计记录不丢失。
- AC13（F12）：可重试错误先执行有限退避；持续失败后按配置切换至下一个模型，并向用户展示原模型、原因和目标模型。
- AC14（F13）：用户级与项目级配置按既定优先级生效；缺失密钥、无效字段和未知协议均产生明确诊断，日志中不出现密钥。
- AC15（F14）：示例外部工具和模型适配器无需修改核心即可注册、执行、取消并经过权限检查。
- AC16（F15、N4）：一次完整运行产生可关联的本地时间线；默认记录中不包含密钥或已识别的敏感字段，且无网络遥测请求。
- AC17（F16）：任务结束报告准确列出文件变化、验证命令和实际结果；测试失败时最终状态不得显示为完全成功。
- AC18（N1、N5、N6）：Linux 上的格式检查、严格类型检查、单元测试、协议契约测试和端到端模拟场景全部通过；没有模型密钥时测试套件仍可完整运行。
