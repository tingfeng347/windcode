# Windcode 临时多智能体并行编排 Spec

## 背景

Windcode 已具备单智能体执行循环、结构化事件、工具调度、权限审批、会话恢复、上下文压缩和本地运行轨迹，但一个任务目前只能由单个智能体串行推进。面对可独立调研、实现或验证的子任务时，单智能体容易受到上下文膨胀、执行耗时和不同工作流相互干扰的限制。

本轮借鉴五类架构的长处：

- Claude Code 风格的角色化委派、权限继承与用户控制
- Codex 的持久化父子关系、显式委派策略和结构化生命周期
- Pi 的独立上下文、有限并发和简洁扩展边界
- GenericAgent 的自包含任务输入与低耦合执行思想
- Hermes 的工具限制、资源预算、取消传播和结果摘要

本轮只实现任务级临时子智能体。长期团队、成员邮箱、共享任务板、跨会话长期记忆及插件生命周期留给后续独立项目。

## 目标

- 支持主智能体把相互独立的子任务并行委派给临时子智能体，并统一汇总结果。
- 每个子智能体使用隔离的新会话和独立执行状态，避免父会话历史及其他子任务互相污染。
- 读取型任务安全共享当前工作区；写入型任务在隔离工作区完成、验证并形成可审查的提交。
- 由父智能体统一检查和集成子智能体的代码成果，冲突或验证失败不得被静默处理。
- 让子智能体的创建、排队、运行、审批、用量、结果、失败、取消和遗留工作区对用户可见且可审计。
- 保持现有单智能体行为和公共开发接口兼容，并为后续长期团队及其他执行后端建立稳定基础。

## 功能需求

- **F1：临时委派**  
  主智能体能够创建一个或一批临时子智能体。每个子智能体只执行一个自包含任务，返回终态结果后结束，不进入长期待命状态。

- **F2：委派触发策略**  
  默认仅当用户明确要求委派、并行或使用子智能体时允许创建；用户级或项目级配置可以启用主动委派。当前生效策略必须可见。

- **F3：扁平拓扑**  
  只有主智能体能够创建子智能体。子智能体不能继续委派，不形成多层智能体树。

- **F4：容量控制**  
  单个父任务默认最多创建 8 个子智能体、同时运行 4 个。超过并发数的任务进入有序队列；超过总数时返回结构化错误。两个限额可以配置，但不能突破系统硬上限。

- **F5：自包含任务**  
  委派内容必须明确描述目标、必要上下文、工作区、约束、预期产物和验证要求。缺少必要字段时不得启动含糊的子任务。

- **F6：上下文隔离**  
  子智能体从全新会话开始，不继承父会话消息、工具调用历史或兄弟任务上下文。它只接收自包含任务，以及运行所需的环境、项目指令、模型默认值和权限上限。

- **F7：能力与权限约束**  
  主智能体可按任务分配角色和最小必要工具集。子智能体不得获得高于父智能体的权限、沙箱或网络能力，也不得绕过父任务的资源限制。

- **F8：读取型任务**  
  读取型子任务可以并行查看父任务对应的工作区快照，但不能修改工作区。发生写入尝试时必须拒绝并返回结构化错误。

- **F9：写入型任务隔离**  
  每个写入型子任务必须基于明确的父提交创建独立工作区和分支。子智能体的修改不得在集成前出现在父工作区中。父工作区不是有效且干净的 Git 工作区时，必须拒绝启动写入型子任务并说明原因；读取型子任务仍可运行。

- **F10：写入型任务交付**  
  写入型子智能体应执行任务要求的验证并提交变更，返回任务摘要、提交标识、变更摘要、验证命令及实际结果。失败或未提交状态必须明确报告。

- **F11：父级集成**  
  父智能体必须先检查子智能体的摘要、变更和验证证据，再把无冲突提交集成回当前分支。发生冲突时停止自动集成，由父智能体处理冲突并重新验证，不得把冲突状态报告为成功。

- **F12：状态与结果顺序**  
  子任务至少具有排队、运行、阻塞、完成、失败和取消状态。实时事件按实际发生顺序展示；批量调用返回的最终结果按原始任务顺序排列。

- **F13：审批与澄清**  
  子智能体需要权限审批时，将请求转发到父会话，并明确展示来源、操作和风险。子智能体不能直接询问用户；信息不足时以 `blocked` 结束，由父智能体统一处理。

- **F14：取消传播**  
  取消父任务时，必须取消正在运行及尚未启动的子任务，并终止其在途模型请求和工具执行。取消某个子任务不得影响其他兄弟任务。

- **F15：清理与异常恢复**  
  已成功集成且无遗留变更的隔离工作区自动清理。未集成、失败、冲突或包含未提交变更的工作区必须保留并报告。进程恢复后可以查看这些记录和路径，但不得自动续跑子智能体或重放副作用。

- **F16：可观测性与审计**  
  系统必须记录父子关系、角色、任务状态、时间、模型用量、工具状态、审批、提交、集成和清理结果，并在终端界面提供可理解的实时状态与最终汇总。

- **F17：公共接口一致性**  
  终端界面和嵌入式开发接口必须共享同一套委派行为与事件语义；外部调用方能够观察子任务事件、响应审批、取消父任务或指定子任务，并取得结构化结果。

## 非功能需求

- **N1：状态隔离**  
  每个子智能体必须拥有独立的会话、上下文、执行控制、工具调用状态和模型用量统计。并行任务不得通过共享可变状态污染其他任务。

- **N2：权限安全**  
  子智能体的有效权限必须是父权限、任务工具集和沙箱约束的交集。配置错误或状态恢复不得造成权限提升、网络放开或工作区越界。

- **N3：Git 安全**  
  系统不得自动提交父工作区修改、重写用户提交历史、强制删除含成果的分支或 Worktree，也不得在冲突时自动选择一方覆盖另一方。

- **N4：响应性**  
  子任务排队、运行和流式事件处理不得阻塞主界面或其他子任务。取消请求应及时传播到模型请求、工具和排队任务。

- **N5：资源有界**  
  系统必须同时限制单个子智能体和整个父任务的模型步骤、工具调用、运行时长、并发数及创建总数，并持续汇总实际资源消耗。

- **N6：持久化可靠性**  
  父子关系、状态转换、提交和集成结果必须以可恢复方式保存。异常中断后不得把未完成任务误判为成功，也不得自动重复模型调用或副作用。

- **N7：顺序确定性**  
  相同输入和模拟执行结果应产生稳定的任务编号、队列顺序和批量结果顺序；并发完成顺序只能影响实时事件时间线，不能改变最终结果对应关系。

- **N8：可诊断性**  
  创建失败、容量超限、权限拒绝、工作区不干净、验证失败、提交失败、集成冲突、取消和清理失败都必须提供结构化类别及可操作说明。

- **N9：兼容性**  
  未启用或未触发多智能体时，现有单智能体、终端界面、开发接口、会话恢复、权限模式和工具行为保持兼容。新增持久化数据必须具备明确版本且不破坏旧会话读取。

- **N10：本地隐私**  
  子任务提示、事件、代码、Diff、提交信息和运行轨迹默认只保存在本地，不因多智能体功能新增遥测或远程同步。

- **N11：离线测试**  
  自动化测试不得依赖真实模型凭据、外部网络或用户现有 Git 仓库；并行、取消、冲突和异常恢复场景必须能在临时仓库和模拟模型中复现。

- **N12：平台范围**  
  首版继续以 Linux 为正式支持平台。写入型子任务要求工作区是有效且干净的 Git 仓库；缺少 Git 或 Worktree 能力时必须明确降级为只支持读取型子任务。

## 不做的事

- 不实现长期驻留的团队成员、成员邮箱、共享任务板、团队创建或团队解散流程。
- 不允许子智能体继续创建子智能体，不实现递归委派或多层协调树。
- 不实现子智能体之间的直接通信、任务转交、抢占或协商；所有结果统一返回父智能体。
- 不在子智能体完成后保留可继续对话的活动实例；需要补充工作时创建新的自包含子任务。
- 不让多个写入型子智能体直接修改同一个工作区，也不允许子智能体绕过独立 Worktree 写入父工作区。
- 不自动快照、提交、暂存或丢弃父工作区中的未提交修改。
- 不自动解决提交冲突，不在冲突时强制覆盖父分支或其他子任务的成果。
- 不为非 Git 工作区提供写入型子智能体的替代隔离方案。
- 不实现跨进程、远程主机、容器集群、tmux 或终端窗格执行后端；首版只提供进程内执行。
- 不实现跨父任务持续运行、崩溃后自动续跑或后台作业队列。
- 不实现 MCP、Skills、Hooks、插件发现、插件安装或完整插件生命周期；这些能力进入后续独立 Spec。
- 不实现跨会话长期记忆、用户画像、经验结晶或自我进化；这些能力进入后续独立 Spec。
- 不把可安装的自定义智能体定义和角色包纳入本轮插件机制；首版角色只作为任务约束和工具配置使用。
- 不改变现有模型协议，不新增专用于多智能体的模型服务。
- 不承诺首版支持 macOS 或原生 Windows 的 Worktree 并行编排。

## 验收标准

- **AC1（F1、F3）**：主智能体可分别创建单个和批量临时子任务；子智能体只执行收到的任务，不能继续委派，并在完成、失败、阻塞或取消后退出。

- **AC2（F2）**：默认策略下，未收到明确委派意图时主智能体不会创建子智能体；收到明确请求时可以创建。启用主动委派配置后可自行创建，当前策略在运行状态中可见。

- **AC3（F4）**：提交 6 个可并行子任务时，最多同时运行 4 个，其余按输入顺序排队；运行中的任务结束后队首任务启动。尝试创建第 9 个子任务时返回容量超限的结构化错误。

- **AC4（F5）**：缺少任务目标、必要上下文、预期产物或验证要求的委派请求被拒绝，并指出缺失内容；完整的自包含任务可正常启动。

- **AC5（F6）**：模拟父会话包含大量历史和兄弟任务私有信息时，子智能体只能看到自己的任务、运行环境和有效项目指令，看不到父会话历史或兄弟任务内容。

- **AC6（F7、N2）**：父任务处于受限权限或受限工具集时，子智能体无法调用越权工具、启用网络或扩大工作区范围；错误配置和恢复后也不会提升权限。

- **AC7（F8）**：多个读取型子任务能够并行读取同一工作区；任一读取型子任务尝试写文件或执行具有写入效果的操作时，被拒绝并产生结构化错误，父工作区保持不变。

- **AC8（F9、N3、N12）**：在干净 Git 仓库中启动两个写入型子任务时，它们获得基于同一明确提交的不同 Worktree 和分支，修改互不可见且父工作区不变；父工作区存在未提交修改、缺少 Git 或不支持 Worktree 时，写任务被拒绝而读任务仍可运行。

- **AC9（F10）**：写入型子任务成功后返回提交标识、变更摘要、验证命令和实际结果；验证失败、提交失败或存在未提交文件时，结果不得标记为成功。

- **AC10（F11）**：无冲突提交在父智能体检查变更和证据后可集成到当前分支并重新验证；构造冲突时自动集成立即停止，父分支不被强制覆盖，最终状态明确显示冲突和待处理事项。

- **AC11（F12、N7）**：用户可观察每个子任务从排队到终态的状态变化；实时事件按实际完成时间出现，而批量最终结果始终与原始任务顺序对应。

- **AC12（F13）**：子智能体触发权限审批时，父会话显示子智能体身份、工具、参数摘要和风险，用户决定正确返回对应子任务；子智能体需要澄清时不直接弹出问题，而是以 `blocked` 结果返回父智能体。

- **AC13（F14、N4）**：取消父任务后，运行中模型请求、工具调用和排队任务均停止，且不再启动后续操作；单独取消一个子任务时，其他兄弟任务继续运行并可正常完成。

- **AC14（F15、N6）**：成功集成且干净的 Worktree 自动删除；未集成、失败、冲突或含遗留变更的 Worktree 被保留并报告。模拟异常退出并恢复后，系统能展示遗留状态，但不会自动续跑任务或重复副作用。

- **AC15（F16、N8、N10）**：一次批量委派产生可关联的本地父子时间线，包含角色、状态、耗时、用量、工具、审批、提交、集成和清理结果；终端界面能实时展示关键状态，且不会新增远程遥测。

- **AC16（F17）**：同一模拟委派任务分别通过终端界面和嵌入式接口运行时，产生语义一致的创建、排队、运行、审批、完成、失败、取消和用量事件；外部调用方可以取消父任务或指定子任务。

- **AC17（N1）**：并发子任务使用相同模拟模型和工具时，会话、消息、工具状态、用量和结果互不串线；一个子任务失败不会破坏其他子任务状态。

- **AC18（N5）**：分别耗尽子智能体模型步骤、工具调用、运行时长以及父任务创建总数时，系统在对应边界终止或拒绝任务，并报告准确的预算类别和累计用量。

- **AC19（N9）**：未启用或未触发多智能体时，现有单智能体端到端场景、旧会话读取、权限模式、终端交互和嵌入式接口行为保持通过。

- **AC20（N11）**：Linux 上的格式检查、静态检查、严格类型检查、单元测试、集成测试和端到端模拟场景全部通过；测试不需要真实模型凭据、外部网络或用户现有 Git 仓库。
