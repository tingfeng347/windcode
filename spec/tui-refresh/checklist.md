# Windcode TUI 启动页与命令菜单 Checklist

> 每一项必须通过运行代码或观察实际渲染验证；实现文件存在不等于验收通过。

## 实现完整性

- [x] 新会话首屏显示居中的 `windcode` ASCII 品牌标识（验证：以新会话启动 TUI，观察首屏品牌名称及位置）。
- [x] 欢迎态显示当前模型、权限模式、沙箱状态和工作区（验证：使用明确模型与权限启动，观察上下文信息与启动参数一致）。
- [x] 欢迎态辅助文案和状态文案均为中文，不出现 Codex、Claude、Pi Agent 或 OpenCode 品牌（验证：检查欢迎态和输入区可见文本）。
- [x] 欢迎态主输入框自动聚焦且可直接提交普通消息（验证：启动后不切换焦点直接输入并按 Enter）。
- [x] 首次提交普通消息后切换为紧凑对话态，大型欢迎标识不再占用消息区（验证：提交“你好”，观察用户消息和助手流式回复区域）。
- [x] 恢复已有会话时直接进入紧凑对话态（验证：使用已有会话启动或通过恢复选择器恢复，观察欢迎标识隐藏）。
- [x] 执行 `/new` 后清空当前消息并重新显示欢迎态（验证：对话后执行 `/new`，观察界面模式和会话状态）。
- [x] 紧凑对话态正确区分用户消息、助手回复、推理完成状态、工具调用、系统消息和错误（验证：运行包含文本、工具和错误事件的模拟任务）。

## 斜杠命令

- [x] 输入首字符 `/` 后立即显示全部现有命令和中文说明（验证：聚焦输入框并按 `/`，不按 Enter）。
- [x] 输入 `/mo` 后菜单只显示匹配的模型命令（验证：继续输入 `mo`，观察候选集合）。
- [x] 输入不存在的命令前缀时菜单隐藏（验证：输入 `/不存在`，观察菜单不可见）。
- [x] 普通文本、含空格的完整命令参数和多行文本不错误触发命令候选（验证：分别输入普通句子、`/model name` 和多行内容）。
- [x] Up/Down 在菜单可见时移动选中行且不越过首尾（验证：在完整菜单中重复按方向键，观察前导选择符号）。
- [x] Tab 将选中命令补全到输入框并保留输入焦点（验证：选择 `/model` 后按 Tab，观察输入内容和焦点）。
- [x] Enter 在菜单可见时执行当前选中命令，不把原始 `/` 当作未知命令提交（验证：选择 `/status` 后按 Enter，观察状态消息）。
- [x] Esc 关闭命令菜单但不清空输入内容，输入框继续聚焦（验证：菜单可见时按 Esc 后继续输入）。
- [x] 提交普通消息、执行命令、创建新会话或恢复会话后菜单均关闭（验证：分别执行四条路径，观察菜单无残留）。

## 视觉与响应式

- [x] 欢迎态只有一个主要视觉焦点，不出现嵌套卡片或装饰性背景图（验证：检查 80×24 和 120×36 渲染快照）。
- [x] 深色背景下正文、弱化状态、选中项和错误信息具有清晰层级（验证：观察欢迎态、菜单态、对话态和错误态快照）。
- [x] 命令菜单位于输入框正上方并共享左侧强调线，不遮挡状态栏（验证：80×24 下输入 `/`，检查组件区域边界）。
- [x] 40×24 下品牌、输入框、命令菜单和状态信息无重叠，命令说明仍可辨认（验证：生成 40×24 菜单态快照并检查区域）。
- [x] 80×24 下欢迎态、命令菜单态和对话态无重叠或异常位移（验证：生成三种状态的 80×24 快照）。
- [x] 120×36 下品牌与输入视觉组保持居中，聊天区充分利用宽度（验证：生成 120×36 欢迎态和对话态快照）。
- [x] 最长可见文本不会覆盖相邻组件，消息流底部不会被输入区遮挡（验证：显示最长命令说明和多行助手回复）。

## 集成

- [x] 命令解析、命令筛选和命令菜单使用同一命令目录（验证：自动化测试逐一比较目录命令与可解析命令）。
- [x] 欢迎态与对话态共用同一个输入组件，切换时输入焦点和键盘绑定保持有效（验证：首次提交后立即输入 `/status`）。
- [x] 模式切换只影响 TUI 展示，不改变 SDK 请求、模型事件或会话记录（验证：运行现有 SDK/TUI 事件语义一致性测试）。
- [x] 流式文本连续更新时命令菜单和输入框保持可交互（验证：模拟慢速增量事件期间输入并关闭命令菜单）。
- [x] 审批和用户提问组件出现时不被命令菜单遮挡，仍可通过键盘操作（验证：运行审批与提问集成测试）。
- [x] 会话恢复仍保留模型上下文，但当前 TUI 不重放旧轮次事件（验证：运行双轮恢复回归测试）。

- [x] 场景 1：启动新会话 -> 看到 `windcode` 欢迎态 -> 输入 `/` -> 使用方向键选择 `/status` -> Enter -> 菜单关闭并显示中文状态信息。
